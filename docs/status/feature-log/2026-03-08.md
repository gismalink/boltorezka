## Voice incident follow-up ("voice не работает")

### Hypothesis and fix

- По runtime-логам signaling path и `call.mic_state` relay работали штатно (`relayedTo=2`), но remote audio мог пропадать при `AudioContext` route.

## Voice incident follow-up: playback route fallback v7

### Change

- В `useVoiceCallRuntime.ts` добавлен аварийный fallback маршрута звука: если `audioElement.play()` не стартует, runtime переключает peer на `AudioContext.destination` (`audioRoute=context`) и поднимает `speakingGain` до текущей громкости.
- В обычном режиме сохранен приоритет `audioRoute=element`; fallback активируется только на фактической ошибке playback retry/resume.
- В `useVoiceRuntimeMediaEffects.ts` обновлена синхронизация gain: `speakingGain` теперь активен только при `audioRoute=context`, иначе принудительно `0` (защита от дублированного звука).

### Rationale

- После перехода на `element-only` route часть браузеров/девайсов могла оставаться без звука при блокировке `element.play()`.
- Точечный fallback возвращает рабочий path старой стабильной логики без полного отката ветки.

### Local validation

- `npm run check` - PASS.
