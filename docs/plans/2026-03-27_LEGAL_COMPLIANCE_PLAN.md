## 0.1) Нормативная база РФ (для этого сервиса)

Точно применимо (must-have):
- [ ] 152-ФЗ `О персональных данных` (оператор ПДн, основания обработки, права субъекта, безопасность, локализация, уведомление РКН).
- [ ] 149-ФЗ `Об информации, информационных технологиях и о защите информации` (режим распространения информации, доступ/ограничения, обязанности владельца инфоресурса).
- [ ] 436-ФЗ `О защите детей от информации, причиняющей вред их здоровью и развитию` (возрастные ограничения и требования к контенту, если сервис доступен детям).

По условиям бизнеса (conditional):
- [ ] Закон РФ `О защите прав потребителей` (если есть платные тарифы/подписка для физлиц).
- [ ] 54-ФЗ `О применении ККТ` (если принимаются онлайн-платежи от физлиц в РФ).
- [ ] Проверка статуса ОРИ по ст. 10.1 149-ФЗ (если функционал и масштаб подпадают под критерии организатора распространения информации).

Примечание:
- Этот план фиксирует продуктово-операционный минимум. Отдельное формальное правовое заключение (legal opinion) выносится в отдельную задачу с профильным юристом.
- Для 18+ доступа в v1 используется риск-ориентированный подход: обязательные предупреждения + технические ограничения + логирование согласия/доступа.

## 5.1) Матрица применимости для Datowave/Datowave

Профиль сервиса (текущий): web + auth (Google/Yandex), пользовательские аккаунты, сообщения/комнаты, cookie-based session.

Решение по применимости:
- [ ] 152-ФЗ: применимо однозначно (есть ПДн аккаунта и событий auth/session).
- [ ] 149-ФЗ: применимо однозначно (информационный ресурс в сети Интернет).
- [ ] 436-ФЗ: применимо условно, определить возрастную модель продукта (18+ либо допуск несовершеннолетних с ограничениями).
- [ ] ЗоЗПП: включается при запуске платного тарифа/подписки.
- [ ] 54-ФЗ: включается при приеме платежей от физлиц.
- [ ] ОРИ (149-ФЗ ст.10.1): вынести в отдельную проверку с юристом по фактическим критериям и масштабу.

## 5.3) Возрастные ограничения для 18+ пространств (NSFW model)

Целевой принцип v1:
- Сервис не маркируется как полностью 18+.
- 18+ доступ включается выборочно: server/room с флагом `nsfw=true`.
- При первой попытке входа в `nsfw=true` пространство показываем age-gate: `Мне есть 18` / `Назад`.

Юридико-продуктовые требования:
- [ ] В Terms/Privacy явно описано, что часть пространств может содержать контент 18+ и доступ к ним ограничен.
- [ ] Перед входом в `nsfw=true` пользователь получает понятное предупреждение о возрастном ограничении.
- [ ] Самодекларация хранится как факт подтверждения доступа, без хранения лишних ПДн (дата рождения не обязательна для v1).
- [ ] Добавлен процесс обработки обращений/жалоб по ошибочному допуску несовершеннолетних.

Технические требования (обязательно):
- [ ] Добавить флаг `nsfw` в модель server/room и API-контракты.
- [ ] Включить backend enforcement: без age-confirm доступ к `nsfw=true` возвращает отказ (`403` + машинный код, например `AgeVerificationRequired`).
- [ ] Проверка распространяется на все входные пути: UI-навигация, direct link, invite link, API вызовы.
- [ ] До age-confirm скрывать/ограничивать предпросмотр 18+ пространств (название/превью/история по политике продукта).
- [ ] Логировать событие подтверждения: `userId`, `spaceId`, `timestamp`, `policyVersion`, `source=self_declaration`.
- [ ] Поддержать отложенный отзыв/пере-подтверждение age-confirm (например, по сроку или по reset настройкам).

Smoke/acceptance для NSFW age-gate:
- [ ] Не подтвержденный пользователь не может войти в `nsfw=true` пространство.
- [ ] Кнопка `Мне есть 18` открывает доступ только к выбранному 18+ пространству (и/или в рамках заданного policy scope).
- [ ] Кнопка `Назад` безопасно возвращает пользователя без утечки 18+ контента.
- [ ] После reload сохраняется ожидаемое состояние age-confirm согласно policy.
- [ ] Deep-link в `nsfw=true` проходит через age-gate и не обходит backend-проверку.
