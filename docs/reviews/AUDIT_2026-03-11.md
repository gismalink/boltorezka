## Найденные проблемы (по приоритету)

### 1) High: входные ID в API не валидируются как UUID до обращения к БД
- Доказательства:
  - `apps/api/src/routes/member-preferences.ts:27-35` разбирает `targetUserIds` как «сырые» строки из query.
  - `apps/api/src/routes/member-preferences.ts:44-45` напрямую использует `ANY($2::uuid[])`.
  - `apps/api/src/routes/member-preferences.ts:68-71` проверяет `targetUserId` в path только на непустое значение.
  - `apps/api/src/routes/member-preferences.ts:97` пишет `targetUserId` в UUID-колонку без строгой валидации.
- Риск:
  - Невалидный UUID может приводить к ошибкам приведения типов в БД и ответам 5xx вместо контролируемого 4xx.
  - Логи засоряются инфраструктурными ошибками, контракт API становится менее предсказуемым.
- Рекомендация:
  - Добавить строгую валидацию через `z.string().uuid()` для `targetUserIds` и `targetUserId`.
  - Возвращать `400 ValidationError` до DB-вызова (fail-fast).

### 2) Medium: auth-токен хранится в localStorage
- Доказательства:
  - `apps/web/src/App.tsx:86` инициализирует токен из localStorage.
  - `apps/web/src/hooks/app/effects/useSessionStateLifecycle.ts:122` сохраняет токен в localStorage.
  - `apps/web/src/hooks/app/effects/useSessionStateLifecycle.ts:128` удаляет токен только при auth-failure.
- Риск:
  - При XSS-уязвимости злоумышленник может прочитать bearer-токен из localStorage.
- Рекомендация:
  - Предпочесть HttpOnly Secure cookie-сессию для веб-клиента.
  - Если bearer-поток пока сохраняется: сократить TTL, добавить ротацию/refresh и усилить CSP.

### 3) Medium: проверка `check` может быть «зеленой» при пропуске ключевых smoke
- Доказательства:
  - `scripts/verify-all.sh:15-45` запускает API/SSO/realtime/web-e2e smoke только через env-флаги.
  - Типичный `npm run check` может завершаться успешно без реальных интеграционных проверок.
- Риск:
  - Ложное чувство стабильности; регрессии попадают дальше по пайплайну.
- Рекомендация:
  - Разделить проверки на:
    - `check:quick` (текущее поведение для локальной быстрой проверки).
    - `check:required` (обязательные smoke-контракты в CI).
  - Завязать CI-gate на `check:required`.

### 4) Medium: тренд на God-module в критичных модулях runtime
- Доказательства:
  - `apps/web/src/App.tsx` = 1804 LOC.
  - `apps/api/src/routes/realtime.ts` = 1774 LOC.
  - `apps/web/src/components/roomsPanel/RoomRow.tsx` = 563 LOC.
  - `apps/api/src/routes/realtime.ts:132+` совмещает lifecycle сокетов, каноническое media-state, метрики и moderation/event-routing.
- Риск:
  - Рост сложности сопровождения, выше риск регрессий, хуже тест-изоляция.
- Рекомендация:
  - Декомпозировать по bounded responsibilities:
    - Realtime: транспорт/сессии отдельно от media-policy и moderation handlers.
    - Web App: выделить orchestration в более мелкие feature-модули.

### 5) Medium: очень ограниченное unit-тест покрытие собственного кода
- Доказательства:
  - При поиске тестов без `node_modules` найден только один файл: `apps/web/src/utils/voiceDiagnostics.test.ts`.
- Риск:
  - Критичная логика опирается в основном на smoke-тесты, локальные регрессии ловятся поздно.
- Рекомендация:
  - В первую очередь добавить unit-тесты для:
    - валидации и ошибок `member-preferences`;
    - логики auto-connect/auto-disconnect голоса;
    - edge-cases парсинга/диспетчеризации realtime envelopes.

## Дополнительные наблюдения
