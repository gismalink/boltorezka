# ADR: End-to-End шифрование сообщений

Дата: 2026-04-04  
Статус: Proposed  
Автор: команда (инициировано в контексте audit/security roadmap)

---

## 1. Контекст

Текущая модель безопасности — **trust-server**:

- Все сообщения передаются через API runtime service в открытом виде (в рамках TLS-туннеля).
- Сервер имеет полный доступ к содержимому чата: читает, хранит, отдаёт клиентам.
- Медиа-трафик (голос/видео) проходит через LiveKit SFU — сервер видит raw аудио/видео треки.
- Текущая защита: HTTPS/WSS (транспорт), HttpOnly Session Cookie (auth), короткий TTL токена с rotation.

Это нормальная модель для корпоративного чата на своём сервере, **но не E2E-шифрование**: если сервер скомпрометирован, атакующий получает все данные.

## 2. Цель этого плана

Определить поэтапный путь к E2E-шифрованию текстовых сообщений (и опционально — медиа) с минимальным нарушением текущей архитектуры и GitOps-процесса.

---

## 3. Что такое E2E-шифрование в данном контексте

E2E (end-to-end) означает: **сервер не может расшифровать содержимое сообщений**.

Это достигается через:
1. Генерацию ключевых пар (публичный/приватный) **на клиентах**.
2. Шифрование сообщений публичным ключом получателя **до отправки на сервер**.
3. Расшифровку **только у получателя** его приватным ключом, который никогда не покидает устройство.

---

## 4. Модели E2E

### Вариант A: Signal Protocol (double ratchet, Olm/Megolm)

Используется: WhatsApp, Signal, Matrix (Element).

- `Olm` — попарный рatchet (1:1 чат).
- `Megolm` — групповой ratchet (комнаты).
- Очень высокая криптографическая безопасность, forward secrecy.

Плюсы:
- Индустриальный стандарт, проверенная криптография.
- Готовые библиотеки (libolm / vodozemac на WASM).

Минусы:
- Высокая сложность реализации key agreement и key backup.
- Нет server-side delivery receipts/search по зашифрованным данным.
- Сложная верификация устройств и управление cross-device ключами.

### Вариант B: MLS (Messaging Layer Security, RFC 9420)

Используется: Google Messages, Cisco Webex (в разработке).

- Более эффективен для групп (TreeKEM), IETF-стандарт.
- Хорошо масштабируется при большом числе участников.

Плюсы:
- Современный стандарт, хорошая групповая масштабируемость.

Минусы:
- Библиотеки менее зрелые (OpenMLS на Rust/WASM — ранняя фаза).
- Требует изменений в server-side (Key Distribution Server).

### Вариант C: Простое E2E (WebCrypto API, ECDH + AES-GCM)

Базовый вариант без advanced ratchet:

- Каждый клиент генерирует ECDH-пару (P-256 или X25519).
- При открытии чата клиент получает публичный ключ собеседника.
- Сообщение шифруется AES-GCM ключом, выведенным через ECDH.
- Сервер хранит зашифрованный blob.

Плюсы:
- Реализуется нативно через WebCrypto API без внешних зависимостей.
- Минимальные изменения в API (payload-поле `encrypted_body` рядом с `body`).
- Хорошо подходит для старта.

Минусы:
- Нет forward secrecy (один ключевой материал на сессию/комнату).
- Нет multi-device (ключ привязан к браузерному хранилищу).
- При потере/смене устройства — история теряется без механизма backup.

## 5. Рекомендуемый путь (поэтапный)

### Этап 0 — Предусловия (текущий statус берём за baseline)

- [x] Транспорт: HTTPS/WSS/TLS везде.
- [x] Auth: HttpOnly Session Cookie + ws-ticket (не bearer в URL).
- [x] CSP и SameSite cookie настроены.
- [ ] Аудит хранения сообщений на сервере (Postgres): что логируется, как долго.
- [ ] Политика retention сообщений: определить, нужно ли хранить историю на сервере вообще.

### Этап 1 — Key Infrastructure (без смены транспорта сообщений)

Цель: создать инфраструктуру управления ключами, не меняя ещё формат сообщений.

Задачи:
- Добавить в API endpoint `GET /v1/users/{id}/pubkey` — возвращает публичный ключ пользователя.
- Добавить endpoint `PUT /v1/me/pubkey` — клиент загружает свой публичный ключ (ECDH P-256 / X25519).
- Хранить в Postgres `user_keys(user_id, public_key, key_id, created_at)`.
- Клиент генерирует keypair через WebCrypto при первом логине и сохраняет приватный ключ в IndexedDB (`CryptoKey`, non-extractable) или exportable для backup.
- Задокументировать модель угроз: кто видит pubkeys, как их доверие верифицируется.

Smoke/тест:
- Unit-тест генерации keypair и шифрования/расшифровки сообщения на клиенте.
- API smoke: загрузка и получение pubkey туда-обратно.

### Этап 2 — Опциональное E2E для DM (Direct Message)

Цель: зашифровать 1:1 переписку (Personal Messages), группой — позже.

Задачи:
- Клиент перед отправкой DM: получить pubkey получателя → ECDH → AES-GCM → зашифровать `body`.
- API принимает поле `encrypted_body` (Base64) + `key_id` + `encryption_version: "ecdh-aes-gcm-v1"`.
- Старый `body` отправляется пустым или не отправляется.
- Клиент-получатель: видит `encryption_version` → расшифровывает через свой приватный ключ.
- Fallback: если у получателя нет pubkey → отправить в plaintext (с индикатором отсутствия E2E в UI).
- UI: иконка замка в строке сообщения — зашифровано/нет.

Smoke/тест:
- E2E-тест: два клиента обмениваются зашифрованным сообщением, сервер не может его расшифровать.
- Smoke: проверка, что сервер хранит только зашифрованный blob.

### Этап 3 — E2E для групповых комнат (Megolm-lite или MLS)

Цель: зашифровать групповой чат в комнате.

Задачи:
- Ввести понятие `room_session_key`: симметричный ключ комнаты, зашифрованный каждому участнику отдельно его pubkey.
- При входе в комнату: клиент получает зашифрованный `room_session_key`, расшифровывает его своим приватным ключом.
- Сообщения шифруются `room_session_key` через AES-GCM.
- При добавлении нового участника: сервер рассылает новую версию `room_session_key`, зашифрованную novому участнику.
- Key rotation: при выходе участника — ротация `room_session_key`.
- Сервер хранит `room_key_versions(room_id, version, encrypted_for_user_id, encrypted_blob)`.

Костыли/ограничения которые принимаем:
- Нет полного forward secrecy на уровне каждого сообщения (упрощённая модель без double ratchet).
- При компрометации `room_session_key` — видны все сообщения с момента создания этой версии ключа.

### Этап 4 — Cross-device и Key Backup

Цель: не терять историю при смене/добавлении устройства.

Варианты:
- **Server-side encrypted backup**: приватный ключ зашифрован passphrase пользователя, хранится на сервере как blob. Сервер не знает passphrase.
- **QR-code / key export на бумагу**: пользователь экспортирует ключ в 12-word seed.
- **Cross-device key agreement**: по протоколу (аналог Matrix "cross-signing").

Задачи:
- Определить политику: если ключ потерян — история недоступна (acceptable) или нужен backup.
- Реализовать минимальный backup: encrypted key blob + passphrase hint на API.

### Этап 5 — E2E для медиа (голос/видео)

Отдельный трек, сложнее текстового E2E.

- LiveKit поддерживает E2E-шифрование медиа через `E2EEManager` (Insertable Streams / WebRTC Encoded Transform).
- Управление ключами аналогично групповому chat: `room_media_key` раздаётся через ту же инфраструктуру.
- Ограничение: Safari поддержка Insertable Streams — только с определённых версий.

---

## 6. Что НЕ меняем и почему

| Компонент | Решение |
|---|---|
| Транспорт (HTTPS/WSS) | Оставляем как есть — TLS остаётся необходим |
| Auth (Session Cookie + ws-ticket) | Не трогаем — сессионная безопасность отдельный слой |
| Server-side search/фильтрация сообщений | После E2E становится невозможна (компромисс) |
| Notifications (push preview) | После E2E push-preview сообщений невозможен без раскрытия ключей push-сервису |
| Сервер-side аудит-лог | После E2E сервер видит только зашифрованные блобы |

## 7. Риски и open questions

1. **Key loss UX**: пользователь теряет устройство → история недоступна. Политика backup обязательна до prod.
2. **Multi-device**: один пользователь с web + desktop + iOS → нужно раздать ключи на все устройства.
3. **Search**: полнотекстовый поиск по зашифрованному контенту становится невозможен. Альтернатива — client-side search по расшифрованному кешу.
4. **Moderation**: на зашифрованные сообщения модераторы не могут заглядывать по определению. Нужна политика.
5. **iOS клиент**: WebCrypto доступна в WKWebView, но KeyStorage через IndexedDB — ограниченная поддержка. Под iOS нативные Keychain нужны.

---

## 8. Метрики успеха

- Этап 1: pubkey API up, клиент генерирует и хранит keypair. Smoke: PASS.
- Этап 2: DM между двумя клиентами шифруется, сервер хранит unintelligible blob. Тест: PASS.
- Этап 3: групповой чат зашифрован, участник после join расшифровывает историю. Тест: PASS.
- Этап 4: пользователь меняет устройство, история восстанавливается через backup. UX-тест: PASS.

---

## 9. Следующий шаг

> Перед началом implementation: согласовать, нужен ли server-side поиск по истории (если да — E2E несовместим),
> и определить политику key backup (безвозвратная потеря или passphrase-backup).

Предлагаемый первый шаг: реализовать **Этап 1** (Key Infrastructure) в отдельной feature-ветке `feature/e2e-key-infra-v1`.
