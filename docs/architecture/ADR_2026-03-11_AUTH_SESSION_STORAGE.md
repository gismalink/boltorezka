# ADR: хранение веб-сессии (localStorage bearer vs HttpOnly cookie)

Дата: 2026-03-11  
Статус: Proposed

## Контекст

Сейчас веб-клиент хранит bearer-токен в localStorage. Это упрощает текущий поток авторизации, но повышает риск компрометации токена при XSS.

Цель: снизить риск утечки сессии без деградации UX и без нарушения текущего GitOps/deploy процесса.

## Варианты

### Вариант A: оставить localStorage bearer (текущий)

Плюсы:
- Минимальные изменения в коде.
- Прозрачная отладка и понятный lifecycle на клиенте.

Минусы:
- Токен доступен JS-коду в рантайме.
- При XSS риск кражи сессии выше.

### Вариант B: перейти на HttpOnly Secure cookie-сессию

Плюсы:
- Токен недоступен из JS (основной выигрыш по безопасности).
- Лучше базовая защита от XSS-эксфильтрации токена.

Минусы:
- Нужно аккуратно настроить cookie policy (SameSite, domain, secure).
- Потребуются изменения в auth middleware и клиентском bootstrap.

### Вариант C: гибридный этап (переходный)

Плюсы:
- Можно мигрировать поэтапно.
- Снижение риска до полного перехода.

Минусы:
- Временное усложнение логики.
- Нужны четкие сроки и план деактивации legacy-механики.

## Решение (предложение)

Принять Вариант C как переходный этап на 1 итерацию:
1. Короткий TTL access-токена + refresh rotation.
2. Усилить CSP и ограничения источников скриптов.
3. Подготовить cookie-сессию как primary режим.
4. После стабилизации переключить web-клиент на HttpOnly cookie (Вариант B).

## План реализации

### Этап 1
- Ввести короткий TTL для access токена.
- Добавить refresh/rotation и серверную инвалидизацию при logout.
- Ограничить хранение чувствительных auth-данных на клиенте.

### Этап 2
- Реализовать cookie-сессию (`HttpOnly`, `Secure`, `SameSite=Lax/Strict` по итогам тестов).
- Перевести auth bootstrap клиента на cookie flow.
- Сохранить fallback только на время миграции.

### Этап 3
- Удалить localStorage bearer flow.
- Закрыть legacy endpoints/ветки кода, если они больше не используются.

## Критерии готовности

- Токен не читается из JS в основном режиме.
- Logout инвалидирует сессию на сервере.
- Smoke auth/sso и realtime проходят на `test`.
- Нет регрессий в входе/выходе и восстановлении сессии.

## Риски и смягчение

- Риск поломки SSO callback: покрыть smoke + ручной сценарий логина.
- Риск кросс-доменных cookie-конфликтов: заранее проверить domain/path/SameSite на test.
- Риск затяжной миграции: ограничить переходный период одной итерацией.

## Примечание

До принятия final-решения по ADR прод-изменения auth storage не выкатывать. Сначала test deploy + smoke + ручная проверка критического login/logout сценария.
